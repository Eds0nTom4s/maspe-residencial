RELATÓRIO TÉCNICO — PROMPT 27
Cursor HMAC + erro padronizado de sync + diff de fila com UPSERT/REMOVE (read-only)
Data: 2026-05-18

1) Resumo executivo
- Cursors de sync passaram a ser assinados com HMAC-SHA256 (payload + assinatura), impedindo manipulação silenciosa de tenant/domain/filtros/lastId/issuedAt.
- Introduzido envelope de erro padronizado para `/device/sync/*` via `SyncErrorResponse` + `DeviceSyncExceptionHandler`.
- Diff incremental da fila ganhou semântica explícita de `updates` com `UPSERT/REMOVE`, reduzindo inferência no client sobre remoção da fila.
- Mantido escopo estrito: tudo permanece read-only; sem POS criando pedido, sem pagamento/callback, sem offline bidirecional.

2) Estratégia de cursor HMAC
- Formato do cursor (v1):
  - `base64url(payloadJson) + "." + base64url(HMAC_SHA256(base64url(payloadJson), secret))`
- Propriedades:
  - `consuma.sync.cursor.hmac-secret`
  - `consuma.sync.cursor.require-signature=true` (default)
- Regras:
  - Cursor adulterado → HTTP 400 com `code=SYNC_CURSOR_INVALID_SIGNATURE`.
  - Cursor legacy sem assinatura é rejeitado quando `require-signature=true`.
  - Cursor não vaza secrets/tokens; não logamos cursor completo.

11) Decisões técnicas tomadas
- Cursor é tratado como input não confiável: assinatura HMAC passa a ser obrigatória por default.
- Erros de sync para device precisam ser previsíveis para app: criado envelope `SyncErrorResponse`.
- Diff de fila deve indicar remoção explicitamente para reduzir heurísticas no client (`UPSERT/REMOVE`).

12) Riscos restantes
- Secret default em ambientes produtivos deve ser proibido/validado por profile (hardening futuro).
- Cursor expirado hoje é tratado por reinício em alguns domínios; o app deve estar preparado para re-sync quando necessário.
