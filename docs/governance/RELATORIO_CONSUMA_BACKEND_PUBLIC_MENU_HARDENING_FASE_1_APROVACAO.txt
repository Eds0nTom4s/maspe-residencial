1. OBJETIVO
================================================================================

Implementar no backend Java/Spring Boot o hardening do cardápio público
definido na Fase 1:

  • Banner customizável por tenant no contexto QR público.
  • Limite máximo de itens por pedido público.
  • Galeria de imagens nos produtos do cardápio público.
  • Vinculação de pedido público a uma sessão de consumo identificada.
  • Gateway SMS TelcoSMS ativado com mock=false e API key fornecida.

2.4 Serviços
--------------------------------------------------------------------------------
  • QrCodeOperacionalService
      - Popula tenantBannerUrl no contexto QR a partir da config do tenant.
      - Exibe maxItensPorPedido no cardápio público.
      - Limita a 4 imagens por produto no cardápio público.

  • PublicQrPedidoService
      - validarLimiteItensPorPedido(Long tenantId, List<...>)
          Soma quantidades; rejeita com BusinessException("LIMITE_ITENS_EXCEDIDO")

      - resolverSessaoParaPedido(..., Long sessaoConsumoId)
          Se sessaoConsumoId informado: valida existência, tenant e status ABERTA.
          Se não informado: mantém comportamento anterior (sessão anônima).

================================================================================
5. DECISÕES TÉCNICAS
================================================================================

  • Limite de imagens por produto no cardápio: 4 (hard limit no service).
    O banco permite N imagens; a API pública expõe no máximo 4.

  • Limite de itens por pedido: opcional (NULL = sem limite).
    Quando configurado e > 0, é aplicado a todos os pedidos públicos do tenant.

  • Sessão identificada: opcional (sessaoConsumoId NULL = comportamento legado).
    Quando informado, a sessão deve pertencer ao mesmo tenant e estar ABERTA.

  • API key SMS: armazenada como default no application.properties com override
    via TELCOSMS_API_KEY, mantendo flexibilidade entre ambientes.
