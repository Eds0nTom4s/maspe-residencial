7. ARQUITECTURA

-> DiscoveryDtoMapper -> DTO HTTP -> controller

contrato HTTP pertence ao backend; transporte Android futuro deve adaptar DTOs ao
dominio Android sem ligar JSON a UiModel.

10. DTOS

11. MAPPERS

PersistentDiscoveryMapper conhece as projections e produz dominio. DiscoveryDtoMapper

12. SERVICE

DefaultDiscoveryService esta marcado @Transactional(readOnly=true), aplica

13. REPOSITORY

DiscoveryRepository define somente home, search e merchant. Nao possui save, update,
delete ou comando. Paginacao, filtros e ordenacao persistentes acontecem no banco.

14. PERSISTENT REPOSITORY

PersistentDiscoveryRepository e o unico bean DiscoveryRepository no runtime normal.
Usa PageRequest no banco, NAME ascendente e slug ascendente como desempate. Search e
Home executam count+select; Detail executa projection principal+localizacao primaria.
Nao carrega todos os merchants para paginar em memoria.

A query e JPQL parametrizada. Query textual nao e concatenada. O valor LIKE e
escapado antes do bind: ! -> !!, % -> !%, _ -> !_. O agregado de produtos faz parte
da projection SQL e nao dispara query por merchant.

15. INMEMORY FAKE

InMemoryDiscoveryRepository nao possui @Repository, @Component, @Service ou @Bean.
E instanciado explicitamente somente por testes para cenarios CONTENT, EMPTY, ERROR e
OFFLINE. OFFLINE e nome interno do fake e converte para SERVICE_UNAVAILABLE; nao e
codigo HTTP publico.

24. PAGINACAO

Backend zero-based: page=0 e a primeira pagina; defaults page=0/pageSize=20;

duplicacoes/omissoes entre paginas.

Conversao Android futura: pagina de dominio 1 -> request 0; pagina N -> N-1;
response 0 -> dominio 1.

ANALISE ARQUITECTURAL

Discovery e read-only porque a descoberta publica consulta visibilidade e capacidade;
comandos pertencem aos bounded contexts administrativos/operacionais. Reutilizar
entidades JPA no HTTP vazaria schema, lazy loading e dados internos. Projections
reduzem superficie e tornam query count previsivel. Uma politica publica centralizada
evita divergencia entre Home, Search e Detail.

GET publico nao recebe tenant porque o utilizador ainda esta antes da seleccao de
operador. Token invalido nao deve bloquear recurso anonimo nem deve autenticar. O
TenantContext e removido para impedir influencia acidental do header/JWT. Merchant
privado retorna 404 generico para nao confirmar existencia.

Sort sem fonte retorna erro para impedir resultados semanticamente falsos. Rating,
distance, popularity, featured e horarios nao sao simulados. Paginacao zero-based
segue PageRequest/Spring Data; Android adapta a sua pagina conceptual 1-based. NAME e
default porque e o unico sort persistente real e deterministico.

no-store e conservador enquanto publicacao/catalogo nao possuem invalidacao ou ETag.
503 usa SERVICE_UNAVAILABLE para separar indisponibilidade backend de dispositivo
Android offline. O Android adapta transporte por DTO+mapper e preserva o seu dominio;
JSON nunca deve alimentar UiModel directamente.

InMemory permanece fake explicito para testes de cenarios, sem bean runtime. A divida
tecnica consciente inclui publication flag, geografia, sinais de ranking, horarios,
cache e IT Discovery PostgreSQL. O impacto futuro deve ser tratado por fonte de dados,
versionamento e contract tests, nunca por fallback silencioso.

Riscos operacionais principais: collation PostgreSQL, mutabilidade de slug, drift de
capabilities e custo de no-store. Criterios para Android integrar: consumir apenas os
tres endpoints, converter pagina, enviar NAME, omitir capacidades sem suporte,
tolerar campos/enums futuros, respeitar catalogAvailable e mapear erros sem colapsar
semanticas.

FRONTEIRAS QUE NAO DEVEM SER QUEBRADAS

- Discovery nao escreve nem recebe tenant do cliente.
- Discovery nao expoe entidades ou dados administrativos.
- Home, Search e Detail usam a mesma politica.
- Android nao recebe JPA nem Retrofit na UI.
- Sort sem suporte nao tem fallback.
- Dados sem fonte nao sao simulados.
- Empty nao e erro.
- Merchant privado nao e distinguivel.
- Backend usa page zero-based; Android adapta.
- catalogAvailable vem do backend.
- Availability desconhecida e UNKNOWN.
- HTTP 503 nao significa dispositivo offline.
- Cache futuro exige invalidacao/freshness explicitas.

CONFORMIDADE FINAL DOS 47 PONTOS

Todos os 47 pontos da auditoria foram confirmados. O unico desvio encontrado foi uma
lacuna de prova: o teste da cadeia real nao exercitava explicitamente bearer com
aparencia valida em GET publico nem bearer invalido em rota privada. Foram adicionadas
duas assercoes ao teste existente, que passou; nenhuma implementacao foi alterada e a
contagem Discovery permaneceu 73.
