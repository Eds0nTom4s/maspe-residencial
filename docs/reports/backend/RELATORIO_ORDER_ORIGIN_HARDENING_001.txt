1. OBJETIVO
-----------
Formalizar a origem do pedido como dado explícito, estável, auditável e usado
pelas policies operacionais, eliminando inferências frágeis.

2. PROBLEMA TRATADO
-------------------
A política operacional já existia, mas a origem do pedido ainda era inferida em
vários pontos (sessão, mesa, QR, template). Isso criava risco de aplicação de
regras erradas de aceite, pagamento, produção e auditoria.

3. ENUM / TIPO CRIADO
---------------------
Arquivo: src/main/java/com/restaurante/model/enums/PedidoOrigem.java

Valores:
  QR_MESA, QR_PRINCIPAL, QR_PUBLICO,
  SESSAO_PARTICIPANTE, SESSAO_CONSUMO,
  OPERADOR_INTERNO, DEVICE_POS, PDV_INTERNO,
  DEVICE_KDS, CAIXA, SISTEMA, LEGADO, UNKNOWN

O enum aninhado OperationalTemplatePolicy.PedidoOrigem foi removido;
todas as referências agora usam o enum top-level.

4. PERSISTÊNCIA / MIGRATION
---------------------------
Arquivo: src/main/resources/db/migration/V75__pedido_origem.sql

ALTER TABLE pedidos ADD COLUMN IF NOT EXISTS pedido_origem VARCHAR(30);
UPDATE pedidos SET pedido_origem = 'LEGADO' WHERE pedido_origem IS NULL;

- Coluna aditiva, nullable, não destrutiva.
- Default para dados legados: LEGADO (conservador).

5. CRIAÇÃO PÚBLICA (QR)
-----------------------
Arquivo: src/main/java/com/restaurante/service/PublicQrPedidoService.java

7. POLICIES INTEGRADAS
----------------------
Arquivo: src/main/java/com/restaurante/service/operacional/OperationalTemplatePolicy.java

- resolvePedidoOrigem: prioriza origem persistida; fallback conservador.
- requiresAcceptanceBeforePayment: QR/public exige aceite; DEVICE_POS/PDV não.
- allowsImmediatePayment: DEVICE_POS e PDV_INTERNO com actor autorizado.
- productionFlow: KDS/QR/REST → REQUIRED; PONTO/DEVICE → OPTIONAL; CAIXA → NONE.
- canAccept / canReject: exclui DEVICE_POS, DEVICE_KDS, PDV_INTERNO.
- canConfirmPayment: exclui TENANT_KITCHEN / DEVICE_KDS.
- canMoveProduction: respeita productionFlow.

Arquivo: src/main/java/com/restaurante/service/operacional/PedidoAllowedActionsService.java
- Usa origem resolvida para calcular allowedActions / actionReasons.

Arquivo: src/main/java/com/restaurante/service/operacional/PedidoStatusTransitionService.java
- Continua usando OperationalTemplatePolicy para aceite/rejeição.
- Registra pedidoOrigem nos eventos de aceite/rejeição/cancelamento/bloqueio.
