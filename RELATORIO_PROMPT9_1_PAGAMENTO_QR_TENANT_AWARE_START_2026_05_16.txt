RELATÓRIO TÉCNICO — PROMPT 9.1 (INICIAR PAGAMENTO POR QR TENANT-AWARE, SEM CALLBACK)
Data: 2026-05-16
Sistema: CONSUMA (Spring Boot / PostgreSQL / Flyway / AppyPay)

1) RESUMO EXECUTIVO
- Implementado início de pagamento digital (AppyPay) para Pedido criado via QR público, com:
  - validação tenant-safe QR + Pedido
  - Pagamento com tenant_id obrigatório (defesa em profundidade)
  - externalReference tenant-aware compacta (<=15 chars)
  - idempotência obrigatória no endpoint de iniciar pagamento (Idempotency-Key)
  - endpoint público para iniciar pagamento sem JWT
- Callback/confirmação do pagamento fica para Prompt 9.2.

2) DECISÃO: TENANT_ID EM PAGAMENTO
- `pagamentos_gateway.tenant_id NOT NULL` com FK para `tenants(id)`.
- Backfill:
  - preferencial: pagamento -> pedido -> tenant_id
  - fallback: pagamento -> fundo -> sessão -> instituição -> tenant
  - fallback final: tenant `LEGACY`

5) IDEMPOTÊNCIA DE INÍCIO DE PAGAMENTO
Entidade/Repo/Service:
- src/main/java/com/restaurante/model/entity/PublicQrPaymentRequest.java
- src/main/java/com/restaurante/model/enums/PublicQrPaymentRequestStatus.java
- src/main/java/com/restaurante/repository/PublicQrPaymentRequestRepository.java
- src/main/java/com/restaurante/service/PublicQrPaymentIdempotencyService.java

Regras:
- Header `Idempotency-Key` obrigatório (body opcional)
- retry com mesma chave/payload retorna mesmo pagamento
- mesma chave com payload diferente retorna 409
- evita dupla chamada ao gateway no mesmo retry idempotente

6) ENDPOINT PÚBLICO
Controller:
- src/main/java/com/restaurante/controller/PublicQrController.java

Endpoint:
- POST /api/public/q/{token}/pedidos/{pedidoId}/pagamentos

7) SERVICE PÚBLICO DE PAGAMENTO POR QR
Service:
- src/main/java/com/restaurante/service/PublicQrPagamentoService.java

Fluxo:
- resolve QR (ativo/não revogado, tenant ATIVO)
- busca Pedido por id + tenantId (tenant-safe)
- valida contexto (instituição/unidade/mesa quando aplicável)
- aplica idempotência de pagamento
- cria Pagamento (tenant + pedido + externalReference) em PENDENTE
- chama AppyPayClient.createCharge(...)
- persiste dados do gateway (chargeId, entity/reference/paymentUrl)
- NÃO confirma pagamento nesta fase
