2.1 VALIDAÇÃO OPERACIONAL DE COZINHA (CRÍTICO) ✅
────────────────────────────────────────────────

Arquivo: src/main/java/com/restaurante/service/PedidoService.java

PROBLEMA:
- Produtos podiam ser pedidos mesmo com cozinha inativa
- Disponibilidade do produto ≠ operacionalidade da cozinha
- Cliente recebia erro apenas após SubPedido criado

SOLUÇÃO:
```java
// Determinar cozinha responsável pelo item
Cozinha cozinha = subPedidoService.determinarCozinha(
    produto.getCategoria(), 
    unidadeConsumo.getUnidadeAtendimento().getId()
);

// ✅ VALIDAÇÃO CRÍTICA: Cozinha deve estar ATIVA
if (cozinha.getAtiva() == null || !cozinha.getAtiva()) {
    throw new BusinessException(
        String.format("Produto '%s' não pode ser pedido: cozinha '%s' está inativa", 
            produto.getNome(), cozinha.getNome())
    );
}
```

IMPACTO:
- ✅ Bloqueia pedidos ANTES de criar SubPedidos
- ✅ Erro de negócio claro para o cliente
- ✅ Previne frustração operacional (pedido criado mas não executável)

```java
// ConfiguracaoFinanceiraSistema.java
@Column(name = "limite_pos_pago", nullable = false, precision = 10, scale = 2)
@Builder.Default
private BigDecimal limitePosPago = new BigDecimal("500.00");

// ConfiguracaoFinanceiraService.java (removida constante)
// ANTES:
private static final BigDecimal LIMITE_POS_PAGO_PADRAO = new BigDecimal("500.00");

// DEPOIS: busca do banco
ConfiguracaoFinanceiraSistema config = buscarOuCriarConfiguracao();
BigDecimal limitePosPago = config.getLimitePosPago();
```

IMPACTO:
- ✅ Limite configurável via frontend admin (sem redeploy)
- ✅ Banco de dados é fonte de verdade
- ✅ Alterações auditadas automaticamente (updatedAt, atualizadoPorNome)

2.3 VALOR MÍNIMO DE OPERAÇÕES (NOVA REGRA) ✅
────────────────────────────────────────────

Arquivos:
- src/main/java/com/restaurante/model/entity/ConfiguracaoFinanceiraSistema.java
- src/main/java/com/restaurante/service/FundoConsumoService.java
- src/main/java/com/restaurante/config/InicializacaoFinanceiraConfig.java
- src/main/resources/application.properties

PROBLEMA:
- Nenhuma validação de valor mínimo em operações financeiras
- Possibilidade de recargas/débitos de valores irrisórios (ex: 0.01 AOA)
- Custo operacional de transações pequenas não justificável

SOLUÇÃO:

**Passo 1: Adicionar campo na configuração**
```java
@Column(name = "valor_minimo_operacao", nullable = false, precision = 10, scale = 2)
@Builder.Default
private BigDecimal valorMinimoOperacao = new BigDecimal("10.00");
```

**Passo 2: Carregar de application.properties (BOOTSTRAP)**
```properties
# application.properties
financeiro.valor-minimo-operacao=10.00
```

**Passo 3: Inicialização automática (primeira execução)**
```java
@EventListener(ApplicationReadyEvent.class)
@Transactional
public void inicializarConfiguracaoFinanceira() {
    var configExistente = configuracaoRepository.findAtual();
    
    if (configExistente.isPresent()) {
        log.info("✅ Configuração JÁ EXISTE. BANCO é fonte de verdade.");
        return;
    }
    
    // PRIMEIRA EXECUÇÃO: criar com valores do properties
    ConfiguracaoFinanceiraSistema config = ConfiguracaoFinanceiraSistema.builder()
        .valorMinimoOperacao(new BigDecimal(valorMinimoOperacaoPadrao))
        .build();
    
    configuracaoRepository.save(config);
}
```

**Passo 4: Validação em operações financeiras**
```java
// FundoConsumoService.recarregar()
BigDecimal valorMinimo = configuracaoFinanceiraService
    .buscarOuCriarConfiguracao()
    .getValorMinimoOperacao();
    
if (valor.compareTo(valorMinimo) < 0) {
    throw new BusinessException(
        String.format("Valor de recarga (%s AOA) abaixo do mínimo permitido (%s AOA)", 
            valor, valorMinimo)
    );
}
```

IMPACTO:
- ✅ application.properties = valor inicial (bootstrap)
- ✅ Banco de dados = fonte de verdade (após primeira execução)
- ✅ Alterável via frontend admin (sem redeploy)
- ✅ Previne transações economicamente inviáveis

2.5 INICIALIZAÇÃO FINANCEIRA AUTOMÁTICA ✅
────────────────────────────────────────

Arquivo: src/main/java/com/restaurante/config/InicializacaoFinanceiraConfig.java

PROBLEMA:
- Configurações financeiras não inicializadas automaticamente
- Risco de NullPointerException na primeira execução
- Valores padrão não documentados

SOLUÇÃO:
```java
@EventListener(ApplicationReadyEvent.class)
@Transactional
public void inicializarConfiguracaoFinanceira() {
    var configExistente = configuracaoRepository.findAtual();
    
    if (configExistente.isPresent()) {
        log.info("✅ Configuração JÁ EXISTE");
        log.info("📌 BANCO é fonte de verdade. properties IGNORADO.");
        return;
    }
    
    log.info("⚠️  PRIMEIRA EXECUÇÃO");
    log.info("📥 Carregando valores de application.properties:");
    
    ConfiguracaoFinanceiraSistema config = ConfiguracaoFinanceiraSistema.builder()
        .limitePosPago(new BigDecimal(limitePosPagoPadrao))
        .valorMinimoOperacao(new BigDecimal(valorMinimoOperacaoPadrao))
        .posPagoAtivo(true)
        .build();
    
    configuracaoRepository.save(config);
    
    log.info("✅ Configuração PERSISTIDA com sucesso!");
    log.info("📌 BANCO é fonte de verdade a partir de agora.");
}
```

COMPORTAMENTO:
1. **Primeira execução:** Carrega de application.properties → persiste no banco
2. **Execuções seguintes:** Ignora application.properties → usa banco

LOGS DETALHADOS:
```

📌 A partir de agora: BANCO DE DADOS é a fonte de verdade.
```
