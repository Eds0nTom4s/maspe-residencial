RELATÓRIO TÉCNICO — PROMPT 19
Gestão mínima de usuários do tenant + auditoria operacional
Data: 2026-05-17

1) RESUMO EXECUTIVO
- Implementada a gestão mínima de equipe do tenant via endpoints `/api/tenant/usuarios`, usando o modelo existente `TenantUser`.
- Suporta multi-role por usuário (modelo interno 1 linha por role) e expõe roles agregadas no DTO.
- Aplicado enforcement de `maxUsuarios` por tenant (contagem por usuários distintos não-REMOVIDO).
- Implementada auditoria operacional mínima (`tenant_audit_logs`) para ações administrativas de gestão de usuários.
- RBAC aplicado: somente `TENANT_OWNER` e `TENANT_ADMIN`; com regras adicionais de política (ADMIN não mexe em OWNER/ADMIN; proteção do último OWNER).

7) SERVICES / POLICY
- `src/main/java/com/restaurante/service/tenantadmin/TenantUsuarioService.java`
  - listar, buscar, criar/vincular, alterar roles, suspender, reativar, remover (remoção lógica)
  - multi-role por usuário: cria/reativa/remove linhas `TenantUser` por role
  - validações: email/telefone, usuário já membro (409), limite, último OWNER, cross-tenant 404
  - compatibilidade com modelo atual de `User` (schema exige `telefone` NOT NULL para criar novo User)
- `src/main/java/com/restaurante/service/tenantadmin/TenantUsuarioPolicy.java`
  - regras OWNER vs ADMIN:
    - ADMIN não pode atribuir OWNER/ADMIN
    - ADMIN não pode modificar/suspender/remover usuários que tenham OWNER/ADMIN
    - OWNER não pode suspender/remover o último OWNER ATIVO
- `src/main/java/com/restaurante/service/tenantadmin/TenantAuditService.java`
  - grava eventos operacionais sem dados sensíveis (não grava senha)

10) ESTRATÉGIA MULTI-ROLE (MODELO INTERNO)
- Mantido o modelo atual: `TenantUser` com `unique(tenant_id, user_id, role)`.
- API agrega roles por usuário e retorna `roles: [TENANT_*]` no `TenantUsuarioResponse`.
- Alteração de roles faz “set”:
  - roles removidas -> `REMOVIDO`
  - roles adicionadas -> cria nova linha
  - roles já existentes -> reativa/ajusta estado para `ATIVO`

11) PROTEÇÃO DO ÚLTIMO OWNER
- Bloqueia:
  - suspensão do último `TENANT_OWNER` com estado `ATIVO`
  - remoção lógica do último `TENANT_OWNER` com estado `ATIVO`

12) AUDITORIA (O QUE REGISTRA / O QUE NÃO REGISTRA)
Registra:
- criação/vínculo e reutilização de usuário
- alteração de roles
- suspensão/reativação/remoção
- bloqueio por limite
Não registra:
- senha temporária
- secrets/tokens
