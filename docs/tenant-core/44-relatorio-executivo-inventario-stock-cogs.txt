PROMPT 44 — RELATÓRIO EXECUTIVO FINAL
CONSUMA / INVENTORY CORE / STOCK / COGS / MARGEM / EVIDENCE

Data: 2026-05-25

1) Objetivo do Prompt 44
- Implementar a base operacional de inventário/stock e custo de mercadoria vendida (COGS), com:
  - movimentos append-only;
  - receitas técnicas (composição);
  - baixa automática por pedido pago (PAYMENT_CONFIRMED);
  - cálculo estimado de COGS e margem;
  - evidência consolidada em Evidence Bundle (inventoryEvidence);
  - auditoria sanitizada.

2) Escopo implementado
- Itens de inventário (InventoryItem) com quantidade atual materializada e custo médio.
- Unidades de medida (UnitOfMeasure) e conversões (UnitConversion) para normalização em base unit.
- Receitas técnicas (InventoryRecipe + InventoryRecipeLine) com yieldQuantity e status.
- Mapeamento produto → política de stock (ProductInventoryMapping).
- Movimentos de stock append-only (InventoryMovement).
- Consumo automático pós-pagamento confirmado:
  - listener AFTER_COMMIT em PaymentConfirmedForFiscalIssueEvent (reuso do evento do Prompt 43.1);
  - criação idempotente por tenantId+pedidoId via InventoryConsumptionRecord;
  - criação de InventoryMovement SALE_CONSUMPTION por ingrediente.
- COGS e margem estimada por pedido:
  - totalCost = soma das linhas consumidas;
  - netRevenueAmount preferencialmente via FiscalDocument (taxable+exempt) quando existir;
  - fallback para Pedido.total se não houver FiscalDocument.
- Evidence Bundle:
  - nova seção inventoryEvidence para bundles novos;
  - bundles antigos continuam legíveis (campo ausente/null).

3) O que NÃO foi implementado (por design / fora do Prompt 44)
- Contabilidade geral, livro razão, lançamentos contabilísticos.
- Inventário fiscal oficial/SAF-T/integração AGT.
- Gestão avançada de compras/fornecedores.
- Emissão/baixa offline definitiva no POS (consumo roda server-side após confirmação/replay).
- Devoluções/estornos com reversão automática de consumo/COGS.
- Múltiplos armazéns complexos; lotes/validade; rastreio sanitário.
- UI frontend.
