RELATÓRIO TÉCNICO — PROMPT 40.4
Replay assíncrono com operationId + progress tracking + rate-limit por tenant
Data: 2026-05-23

1) RESUMO EXECUTIVO
- O replay controlado (Prompt 40.3) foi evoluído para execução assíncrona rastreável por `operationId`.
- O submit cria uma operação `PENDING` + itens por comando elegível; o worker processa em batch e atualiza progresso/contadores.
- Há rate-limit por tenant no submit (operações ativas e por hora) e lock para evitar replay concorrente do mesmo comando.
- Progress events possuem rate-limit (delta percentual + intervalo mínimo), com eventos finais sempre emitidos.
- Re-run seguro permite reprocessar itens (FAILED/BLOCKED/PENDING) sem tocar em SUCCEEDED/NOOP.

9) COMO O SUBMIT ASYNC FUNCIONA
- Admin envia filtros (statuses/types/ids) + `reason` obrigatório (+ `force` opcional).
- Serviço:
  - valida tenant/RBAC e rate-limit;
  - resolve `serverSyncId` tenant-safe;
  - avalia elegibilidade (Prompt 40.3);
  - cria operação `PENDING` e itens `PENDING`;
  - marca comandos com `replay_in_progress=true` (impede concorrência).
- Resposta retorna `operationId` para polling.

10) COMO OPERATIONID / ITEMS FUNCIONAM
- `operationId` (UUID) é o identificador público estável.
- Cada item referencia um `device_offline_command_id` (com payload imutável).
- Item mantém estado e retry/backoff independente.

11) PROGRESS TRACKING
- `progressPercent = (succeeded + noop + blocked + failed) / total * 100`
- Counters recalculados pelo worker a cada batch/itens.

12) RATE-LIMIT POR TENANT
- No submit:
  - bloqueia acima de `max-active-operations-per-tenant`
  - bloqueia acima de `max-operations-per-tenant-per-hour`
  - bloqueia submit concorrente para o mesmo `serverSyncId` com operação `PENDING/RUNNING`
- Evento `DEVICE_OFFLINE_REPLAY_RATE_LIMITED` registrado (sanitizado).

13) replayInProgress (CONCORRÊNCIA)
- No submit, cada comando selecionado recebe `replay_in_progress=true` + `current_replay_operation_id`.
- O worker limpa o lock do comando ao terminar o item (e limpa tudo ao finalizar a operação como safety net).
- Mantém o replay individual/síncrono protegido contra concorrência (o comando fica marcado como “em replay”).

14) RE-RUN SEGURO
- `POST .../rerun`:
  - bloqueia se operação está `PENDING/RUNNING`
  - reseta apenas itens nos statuses informados (default recomendado: FAILED/BLOCKED)
  - não reseta SUCCEEDED/NOOP
  - incrementa `retry_count`

15) SEGURANÇA / RBAC / CROSS-TENANT
- Submit/rerun: apenas `TENANT_OWNER`/`TENANT_ADMIN`.
- Status/items: `TENANT_OWNER`/`TENANT_ADMIN`/`TENANT_FINANCE`.
- Cross-tenant protegido por `tenantId` via `TenantContext` (404 para recursos fora do tenant).
- Regras do replay (40.3) continuam garantindo:
  - não reprocessar APPLIED/DUPLICATE/IDEMPOTENCY_CONFLICT;
  - revalidação por handlers (capability/policy/turno/estado de entidades);
  - payload/clientRequestId/payloadHash imutáveis.

21) DECISÕES TÉCNICAS TOMADAS
- Reutilização do core de replay do Prompt 40.3 (`TenantOfflineSyncReplayService.replayOne`) para evitar divergência de regras.
- Worker com claim por item via `FOR UPDATE SKIP LOCKED` + backoff por item.
- `replay_in_progress` no comando como lock simples contra concorrência entre operações/replays.
- Progress audit com rate-limit para reduzir ruído, mas eventos finais sempre emitidos.

22) RISCOS RESTANTES
- Operações muito grandes ainda podem gerar custo DB alto (recompute counters via múltiplos counts). Otimizável com contadores incrementais.
- Sem cancelamento avançado, um tenant pode precisar esperar itens expirarem/backoff ou usar rerun.
- Em cluster multi-instância, locks dependem de `locked_at` e timeout; ajuste fino pode ser necessário em produção.
