RELATORIO TECNICO — PROMPT 40.3 (REPROCESSAMENTO/REPLAY CONTROLADO POR SYNC SESSION + EXPORT DIAGNOSTICO SANITIZADO)
Data: 2026-05-23
Projeto: CONSUMA / Sistema de Restauração (Spring Boot + PostgreSQL + JPA/Hibernate + Flyway)

1) RESUMO EXECUTIVO
- Implementado replay controlado (tenant/admin) para comandos offline associados a uma sync session (`serverSyncId`), com:
  - avaliação de elegibilidade (preview);
  - replay por sessão (por statuses/commandIds) com reason obrigatório;
  - replay por comando individual;
  - histórico de tentativas (`device_offline_command_replay_attempts`);
  - export sanitizado de diagnóstico por sync session (JSON).
- Regras centrais preservadas:
  - não altera `clientRequestId`;
  - não altera `payloadHash`;
  - reaplica usando o payload persistido (`payload_json`);
  - APPLIED/DUPLICATE nunca reprocessam;
  - IDEMPOTENCY_CONFLICT nunca reprocessa;
  - AppyPay/OTP/fecho/snapshot/evidence continuam proibidos (já bloqueados no allowlist do offline).
- Replay revalida capability/policy/turno/estado atual via os mesmos handlers de offline sync (sem duplicar lógica financeira).

5) SERVICES
5.1) Elegibilidade
- src/main/java/com/restaurante/service/tenant/offline/DeviceOfflineReplayEligibilityService.java
  - Regras MVP:
    - status replayável: FAILED/CONFLICT/REJECTED
    - nunca: APPLIED/DUPLICATE/IDEMPOTENCY_CONFLICT
    - se `createdEntityType/id` já preenchido => bloqueia
    - se `dependsOnClientRequestId` existe => só elegível se dependência está APPLIED/DUPLICATE no mesmo device
    - REGISTER_LOCAL_ACTIVITY => LOW_VALUE_NOOP (não replay por default)
    - CONFIRM_MANUAL_PAYMENT: se ordem já CONFIRMADA => bloqueia

5.2) Replay + Export
- src/main/java/com/restaurante/service/tenant/offline/TenantOfflineSyncReplayService.java
  - preview(serverSyncId,...)
  - replaySession(serverSyncId,...)
    - cria operationId
    - ordena por commandIndex
    - constrói DevicePrincipal do device alvo a partir das capabilities persistidas (device_operational_capabilities)
    - para cada comando:
      - cria attempt (REQUESTED -> RUNNING)
      - resolve dependsOn/localRef (mesmo device)
      - reescreve payload para ids reais (pedidoId/ordemPagamentoId) quando aplicável
      - chama `DeviceOfflineCommandProcessor.process(...)` com idempotencyKey derivada: `offline:{deviceId}:{clientRequestId}`
      - em sucesso: atualiza comando original para APPLIED + incrementa replayCount + lastReplayAttempt*
      - registra eventos
  - replaySingleCommand(commandId,...)
  - exportDiagnostic(serverSyncId)
    - retorna session summary + commands (sanitizados) + attempts (sanitizados)

14) ITENS NAO COBERTOS (POR ESCOPO)
- Não foi implementado export em PDF/Excel (apenas JSON sanitizado).
- Não foi implementado “editar payload” antes do replay (explicitamente proibido).
- “force=true” não promove reprocessamento automático de casos REQUIRES_MANUAL_REVIEW (MVP conservador).
- Não foi implementado replay assíncrono/delayed (execução é síncrona por request).

15) DECISOES TECNICAS
- “Modelo híbrido”:
  - replay attempt guarda histórico completo
  - comando original é atualizado para o estado final quando o replay aplica com sucesso
- Lock pessimista por comando (`findForUpdateById`) para evitar replay concorrente do mesmo comando.
- DevicePrincipal de replay é reconstruído a partir das capabilities persistidas do device (DeviceCapabilityBootstrapService + DeviceOperationalCapabilityRepository).
- IdempotencyKey derivada preservada: `offline:{deviceId}:{clientRequestId}`.

16) RISCOS RESTANTES
- Em cenários de alto volume, replay em massa pode gerar request longa (não async).
- Elegibilidade é conservadora; alguns casos podem exigir revisão manual/ajustes futuros.
