6. Correcoes aplicadas

- repositories PostgreSQL passaram a usar cast explicito nos filtros opcionais nulos:
  - `TurnoOperacionalRepository`
  - `DispositivoOperacionalRepository`
  - `TenantRepository`
  - `OperationalEventLogRepository`
- `TenantPaymentMethodBootstrapService` foi endurecido em dois caminhos:
  - `ensureDefaults(Long tenantId)` com `REQUIRES_NEW` para cenarios que precisam escapar de read-only
  - `ensureDefaultsInCurrentTransaction(Tenant tenant)` para bootstrap do tenant ainda nao commitado
- `BusinessTemplateProvisioningSupport` passou a usar o caminho em transacao corrente no provisioning
- `PaymentMethodPolicyTemplateBootstrapService` passou a usar `REQUIRES_NEW`
- `PagamentoCallbackService` recebeu `noRollbackFor = InvalidCallbackSignatureException.class`
- `PagamentoCallbackInvalidSignatureIT` foi alinhado com:
  - endpoint real `/pagamentos/callback`
  - fixture funcional de produto/categoria
  - propriedades AppyPay minimas para bootstrap do contexto
- `PlatformObservabilidadeControllerIT` teve expectativa ajustada de `400` para `403`
- `PaymentPolicyTemplateRegressionIT` passou a usar `@WithMockUser(username = "owner")`

7. Regressao lateral descoberta e resolvida

- a primeira forma da correcao read-only em `TenantPaymentMethodBootstrapService` usou apenas
  `REQUIRES_NEW`, o que resolveu o caso de escrita sob read-only, mas quebrou o provisioning da
  008-B quando o tenant ainda nao havia sido commitado
- sintoma observado na rerodada da 008-B:
  - `PlatformBusinessTemplateControllerIT`
  - `201` esperado, `404` obtido
- causa raiz:
  - transacao interna nao enxergava o tenant recem-criado
- resolucao final:
  - manter o caminho `REQUIRES_NEW` para o caso read-only
  - adicionar caminho `ensureDefaultsInCurrentTransaction(Tenant tenant)` para o provisioning
