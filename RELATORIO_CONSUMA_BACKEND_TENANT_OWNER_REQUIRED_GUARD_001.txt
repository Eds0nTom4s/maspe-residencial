==================================================
1. OBJETIVO
==================================================

Impedir no backend que:
 - Um tenant seja provisionado sem Owner/Tenant Admin associado.
 - Um tenant seja provisionado sem BusinessAccount.
 - O Platform Admin seja usado como owner/TenantUser OWNER do tenant.
 - O Platform Admin seja usado como BusinessAccountMember OWNER.
 - Credenciais temporárias sejam atribuídas ao Platform Admin como se fossem dele.
 - O reset de senha de tenant seja aplicado a um usuário Platform Admin.

==================================================
2. REGRA DE DOMÍNIO IMPLEMENTADA
==================================================

Tenant sem Owner/Admin é bloqueado?
SIM

Tenant sem BusinessAccount é bloqueado?
SIM (já era bloqueado; validação mantida e reforçada no DTO com @AssertTrue)

Platform Admin é impedido de ser owner do tenant?
SIM

Platform Admin é impedido de ser owner da BusinessAccount?
SIM (via guard: Platform Admin não recebe TenantUser OWNER nem BusinessAccountMember OWNER)

Credenciais temporárias são geradas para owner real?
SIM

/auth/tenants do owner real retorna o tenant?
SIM (validado por TenantOwnerRequiredGuardIT + TenantProvisioningAccessIT)

/auth/tenants do Platform Admin não trata tenant como negócio próprio?
SIM (Platform Admin não recebe TenantUser vinculado via provision-with-access)

/platform/** continua permitindo governança da plataforma?
SIM

Dados legados sem owner não quebram o sistema?
SIM (guards atuam apenas nos novos provisionamentos; dados existentes intactos)

==================================================
8. SEPARAÇÃO PLATFORM ADMIN vs OWNER
==================================================

Platform Admin (ROLE_ADMIN) aparece como:
  - provisionedBy (campo em BusinessAccount via resolveProvisionedBy())
  - ator em logs (PASSWORD_RESET_BY_PLATFORM_ADMIN)

Platform Admin NUNCA é:
  - TenantUser OWNER
  - BusinessAccountMember OWNER
  - Destinatário da senha temporária
  - ownerUserId no response

O response de provisionamento continua expondo ownerUserId/ownerUsername/temporaryPassword
referentes ao owner real, não ao Platform Admin.

14. LIMITAÇÕES
==================================================

1. A guard assertOwnerIsNotPlatformAdmin() é mais eficaz quando o Platform Admin
   possui um User real no banco (para verificação por email/telefone).
   Em contextos @WithMockUser (sem JwtPrincipal), a verificação por identidade
   recorre apenas ao username da autenticação Spring.

2. O cenário de "dual-role" (pessoa física que é dona de negócio E tem acesso admin)
   não é suportado nesta fase — é bloqueado por segurança conforme especificação.

3. Backfill de tenants legados sem owner não foi implementado (out-of-scope).
