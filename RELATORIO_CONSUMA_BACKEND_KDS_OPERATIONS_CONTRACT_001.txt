1. Objetivo

Expor contrato backend tenant-scoped para operacao KDS cobrindo unidades de producao,
fila de subpedidos, detalhe, filtros, transicoes operacionais, controle otimista por
version, isolamento multi-tenant e compatibilidade com fluxos PONTO e RESTAURANTE.

2. Escopo entregue

* Endpoints novos:
  - GET /tenant/kds/unidades-producao
  - GET /tenant/kds/subpedidos
  - GET /tenant/kds/subpedidos/{id}
  - PATCH /tenant/kds/subpedidos/{id}/iniciar-preparo
  - PATCH /tenant/kds/subpedidos/{id}/pronto
  - PATCH /tenant/kds/subpedidos/{id}/entregar
* Contrato de resposta:
  - subPedidoId, pedidoId, pedidoNumero, status
  - unidadeProducaoId, unidadeProducaoNome
  - clienteNome, clienteTelefone
  - mesaId, mesaNome, sessaoId
  - itens com produtoId, produtoNome, quantidade e observacao
  - createdAt, updatedAt e version
  - summary com pendentes, emPreparacao, prontos, entregues e total
* Regras:
  - acesso exige token tenant-scoped com role TENANT_OWNER, TENANT_ADMIN,
    TENANT_OPERATOR ou TENANT_KITCHEN
  - PONTO sem sessao/mesa retorna campos nulos sem quebrar o contrato
  - RESTAURANTE com sessao/mesa retorna os campos quando presentes
  - cross-tenant retorna 404
  - version obsoleta retorna 409 com estado atual

10. Bugs encontrados e correcoes

1)
* descricao:
  - O primeiro criterio de permissao KDS rejeitava tokens tenant-scoped derivados de platform admin
    mesmo apos selecao explicita de tenant no sandbox.
* impacto:
  - endpoints /tenant/kds retornavam 403 no container para operador autenticado com tenantId e role TENANT_ADMIN.
* correcao:
  - regra ajustada para exigir tenantId e role tenant autorizada, sem bloquear o marcador platformAdmin
    quando o token ja esta tenant-scoped.
* validacao:
  - tenant select tenantId 2/3 => 200
  - GET /api/tenant/kds/unidades-producao com token tenant => 200
  - endpoint sem token continua 401
