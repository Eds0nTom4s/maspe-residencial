6) Estratégia de segurança
- Todos endpoints `/tenant/**` usam:
  - TenantContext obrigatório
  - membership ativo no tenant (TenantGuard.assertCurrentUserBelongsToTenant)
  - tenant ativo (TenantGuard.assertTenantActive)
- Não aceitam `tenantId` via query para tenant-admin.
- Os endpoints seguem o padrão do projeto de usar `server.servlet.context-path=/api` (logo `/api/tenant/**` em runtime).

7) Estratégia anti-cross-tenant
- Para recursos por `id`, sempre busca `id + tenantId` (ou navegação indireta Mesa→UA→Instituição→Tenant).
- Se não pertencer ao tenant, retorna 404 com mensagem genérica.

8) Estratégia de QR tenant-admin
- Listagem e detalhe são tenant-scoped.
- Revogação é idempotente (se já revogado, retorna estado atual).
- Geração por mesa:
  - valida mesa no tenant
  - aplica limite `maxQrCodes` via TenantLimitService
  - se já existe QR ativo para mesa, retorna o existente (evita duplicação).
- `qrUrlPublica` montado via `consuma.public-base-url` + `/q/{token}`.

9) Estratégia de pedidos tenant-admin
- Listagem paginada com filtros via query tenant-scoped.
- Detalhe carrega itens + subpedidos em query tenant-scoped.
- Nesta fase: apenas leitura (não altera status operacional/financeiro).
