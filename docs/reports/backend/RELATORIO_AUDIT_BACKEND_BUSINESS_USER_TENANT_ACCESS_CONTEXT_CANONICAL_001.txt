3. PERGUNTA CENTRAL E RESPOSTA EXECUTIVA
-------------------------------------------------------------------------------

O backend possui a arquitectura certa para recuperar o Tenant Select existente:
User -> TenantUser activo -> Tenant activo -> GET /api/auth/tenants -> POST
/api/auth/tenant/select -> JWT TENANT -> /api/tenant/**.

O provisioning canónico inicial também fecha a ligação principal: o Owner
activo da BusinessAccount é enviado ao template como existingUserId e torna-se
TenantUser TENANT_OWNER ATIVO dentro da mesma transacção que cria o Tenant.
Cada novo Business provisionado para a mesma Account recebe novamente esse
Owner como TENANT_OWNER.

Contudo, o contrato não está pronto para recovery produtivo sem hardening:

1. TenantUser é multi-role por desenho e schema (unique por tenant,user,role),
   mas TenantTokenService trata uma membership como singular. GET duplica um
   Tenant por role e POST select pode falhar com resultado não único.
2. replace Owner governa somente BusinessAccountMember/responsavel. Não remove
   ou rebaixa o TenantUser antigo, não cria TenantUser para o novo Owner nos
   Businesses existentes e não incrementa accessVersion.
3. manager add/remove/suspend não propaga nem revoga TenantUser. A concessão
   operacional é explícita apenas durante provisioning ou pela administração
   tenant; a revogação empresarial não revoga acesso operacional já concedido.
4. GET/select e o TenantResolver validam Tenant e TenantUser, mas não estado da
   BusinessAccount nem subscrição. Uma Account suspensa com Tenant ainda ATIVO
   continua seleccionável e operável.

Decisão: CENÁRIO B — BACKEND PRECISA DE HARDENING PEQUENO. O contrato actual é
reutilizável; não se recomenda uma segunda arquitectura nem novos endpoints.

5. AUTHCONTROLLER E CONTRATOS EXACTOS
-------------------------------------------------------------------------------

O context path é /api. Os mappings do controller são relativos a /auth.

POST /api/auth/jwt/login
- público;
- request JSON: username string obrigatório 3..120; password obrigatório com
  mínimo 6;
- response: ApiResponse<AuthResponse>, com accessToken, refreshToken,
  tokenType="Bearer", expiresIn, username e roles;
- emite access JWT GLOBAL no login normal;
- credenciais inválidas são expostas genericamente como BusinessException/400.

GET /api/auth/tenants
- JWT válido e Authentication obrigatórios; @PreAuthorize e SecurityConfig;
- não requer TenantContext e o TenantContextFilter faz skip exacto deste path;
- request: sem body e sem tenant header necessário;
- response: ApiResponse<List<AuthTenantOptionResponse>>;
- item: tenantId Long, tenantCode String, slug String, nome String, estado
  String, ativo Boolean, roles List<String>, templateCode String opcional,
  planoCodigo String opcional, principal Boolean;
- utilizador sem membership elegível: 200 e data=[];
- sem/invalid/expired token: 401.

POST /api/auth/tenant/select
- JWT válido e Authentication obrigatórios;
- request JSON exacto: {"tenantId": Long obrigatório};
- response: ApiResponse<SelectTenantResponse>;
- data: userId, tenantId, tenantCode, slug, nome, tenantNome, tokenType
  "Bearer", accessToken, expiresIn em segundos e roles Set<String>;
- tenantId ausente/malformado: 400;
- Tenant inexistente ou não ATIVO: actualmente BusinessException/400;
- membership não activa/inexistente: 403 para business user;
- Platform Admin pode seleccionar qualquer Tenant ATIVO e a assunção é
  auditada por PLATFORM_TENANT_CONTEXT_ASSUMED.

SecurityConfig protege também as variantes sem context path visíveis ao
servlet: /auth/tenants e /auth/tenant/select.

6. TENANTTOKENSERVICE
-------------------------------------------------------------------------------

Resolução do User:
- principal User: usa directamente;
- principal JwtPrincipal com userId: recarrega User por ID;
- outro principal: 400 "Principal inválido para seleção de tenant".

Platform Admin:
- reconhecido por User.isAdmin() ou authority ROLE_ADMIN;
- GET lista todos os Tenants ATIVOS, independentemente de TenantUser;
- atribui visualmente TENANT_ADMIN e principal=true apenas ao primeiro;
- POST emite tenant JWT com TENANT_ADMIN, status ATIVO, accessVersion=1 e
  platformAdmin=true.

Business user:
- GET usa findActiveTenantOptionsByUserId(userId, ATIVO, Tenant ATIVO);
- POST exige Tenant ATIVO e TenantUser ATIVO;
- roles vêm da membership;
- planoCodigo é apenas o Plano da Subscricao ATIVA, ou null.

Não há validação de BusinessAccountEstado, BusinessAccountMember, link da
Account, Subscricao para elegibilidade, vertical ou tokenType GLOBAL.

Falha multi-role comprovada estaticamente:
- V68 garante apenas UNIQUE(tenant_id,user_id,role);
- TenantUsuarioService representa cada role numa linha e aceita Set de roles;
- GET transforma cada linha em uma opção, duplicando tenantId;
- POST chama findByTenantIdAndUserIdAndEstado como Optional<TenantUser>;
- com duas linhas ATIVAS a query singular pode lançar resultado não único;
- o JWT/select response carregaria apenas uma role mesmo se não falhasse.

-------------------------------------------------------------------------------

BusinessAccountMember possui unique(account,user), role e estado próprios.

BusinessAccountRole real: OWNER, ADMIN, BILLING_MANAGER, MEMBER, VIEWER.
BusinessAccountMemberEstado: ATIVO, PENDENTE, SUSPENSO, REMOVIDO.

Governance garante um único OWNER activo coerente com responsavel_user_id na
Account, mas o conceito é administrativo/comercial. Não é a membership

Matriz real BusinessAccount Role -> TenantUser Role:

ADMIN | NONE; pode ser explicitamente TENANT_ADMIN ou TENANT_FINANCE no preview
BILLING_MANAGER | NONE; pode ser explicitamente TENANT_FINANCE no preview
MEMBER | NONE; não autorizado como acesso adicional canónico

10. PRINCIPAL OWNER
-------------------------------------------------------------------------------

POST /platform/business-accounts cria/associa User, BusinessAccount RASCUNHO,
responsavel e BusinessAccountMember OWNER ATIVO. Não cria TenantUser porque não
há necessariamente Tenant.

No provisioning, toTemplateRequest usa account.getResponsavel().getId() como
owner.existingUserId. Os templates recusam User inexistente/inactivo e criam
TenantUser TENANT_OWNER ATIVO para esse mesmo User.

Resposta: sincronização OWNER -> TenantUser é PARCIAL: existe no provisioning
inicial/futuro, não é mantida após mudanças de governance.

11. MANAGERS
-------------------------------------------------------------------------------

POST /platform/business-accounts/{id}/managers apenas faz upsert de
BusinessAccountMember e incrementa a version da Account. Não cria TenantUser em
Businesses existentes nem futuros automaticamente.

Para um Business futuro, um Manager só recebe acesso se for indicado em
acessos.additionalAccesses no preview. ADMIN pode receber TENANT_ADMIN ou
TENANT_FINANCE; BILLING_MANAGER apenas TENANT_FINANCE. MEMBER/VIEWER são
recusados. Cada Business requer decisão explícita.

Suspender/remover/rebaixar Manager na Account não altera TenantUser anteriormente
concedido e não incrementa tenant_user_access_versions. Portanto GET
/auth/tenants pode continuar igual e um tenant JWT já emitido pode continuar
válido enquanto a TenantUser permanecer ATIVA.

12. OWNER REPLACEMENT
-------------------------------------------------------------------------------

BusinessAccountGovernanceService.replaceOwner:
- rebaixa o OWNER empresarial anterior para ADMIN;
- cria/actualiza o novo BusinessAccountMember como OWNER ATIVO;
- muda responsavel_user_id;
- grava governance event OWNER_REPLACED;
- não consulta Tenants/TenantUsers;
- não remove/rebaixa TENANT_OWNER antigo;
- não cria TENANT_OWNER para o novo Owner em Businesses existentes;
- não incrementa accessVersion de nenhum utilizador.

Consequência: Owner A continua a ver/seleccionar os Businesses antigos; Owner B
não os vê. Businesses provisionados depois da troca usam Owner B. O readiness
dos antigos continua verde porque encontra Owner A. Risco HIGH.

13. PROVISIONING
-------------------------------------------------------------------------------

Fluxo canónico:
PlatformCanonicalBusinessProvisioningController.preview
-> BusinessProvisioningService.preview
-> valida Account/owner/capacidade/plano/vertical/acessos
-> persiste preview sem efeitos operacionais.

Controller.provision
-> regista/adquire lease da BusinessProvisioningOperation
-> executeProvision em TransactionTemplate
-> BusinessTemplateService.provision
-> template PONTO ou REST
-> cria Tenant, Subscricao ATIVA, Instituicao, Unidade, recursos do template,
   User owner reutilizado e TenantUser TENANT_OWNER ATIVO
-> cria acessos adicionais explícitos
-> força Tenant RASCUNHO no contrato canónico
-> marca operation SUCCEEDED/effectsCommitted.

O efeito é atómico: erro faz rollback das entidades operacionais; a operação é
marcada FAILED em transacção separada. Replay SUCCEEDED não recria efeitos.

16. ACTIVATION LIFECYCLE
-------------------------------------------------------------------------------

Account activation valida owner empresarial e muda apenas Account RASCUNHO ->
ATIVA com evento. Não cria/modifica TenantUser.

Business activation calcula readiness e muda apenas Tenant RASCUNHO -> ATIVO com
evento. Não cria/modifica TenantUser. O acesso já deve existir desde provisioning.

Suspensão/bloqueio/cancelamento posterior da Account pelo endpoint de estado não
cascateia Tenant nem TenantUser. Selector/resolver não consultam Account.

17. ONBOARDING LIFECYCLE
-------------------------------------------------------------------------------

Create onboarding: cria somente intenção persistente.
Approval EXISTING: escolhe Account existente e valida owner invariant.
Approval CREATE_NEW: delega a BusinessAccountGovernanceService.createForOnboarding,
criando User se pedido, Account e OWNER empresarial, mas não TenantUser.
Operation link: associa uma BusinessProvisioningOperation existente.
Complete: exige operation SUCCEEDED/effectsCommitted e Tenant RASCUNHO; associa
o resultado ao onboarding, sem activar.

Nenhuma fase de onboarding cria TenantUser directamente. O TenantUser aparece
dentro da operação canónica de provisioning que o onboarding referencia.

19. POST /AUTH/TENANT/SELECT
-------------------------------------------------------------------------------

Valida autenticação, tenantId, existência e Tenant ATIVO. Business user precisa
TenantUser ATIVO. A escolha é por Tenant, não BusinessAccount, correctamente.

Não valida BusinessAccount ATIVA, Subscricao ATIVA ou tokenType GLOBAL. Um JWT
TENANT autenticado também pode chamar select para outra membership do mesmo User.

Após sucesso gera JWT TENANT. A resposta não inclui refresh token. O cliente
deve conservar o token GLOBAL para voltar ao selector e usar o token TENANT nas
APIs operacionais.

20. JWT GLOBAL
-------------------------------------------------------------------------------

Login: POST /api/auth/jwt/login.
Claims modernos: sub, iat, exp, roles CSV, userId, tokenType=GLOBAL.
Uso: autenticação pré-tenant, GET /auth/tenants e POST /auth/tenant/select.

Observação: refresh usa generateToken(username), que emite formato legacy sem
userId/roles/tokenType. O fallback legacy do JwtAuthenticationFilter aceita-o
por default e recarrega User, mas a compatibilidade depende de
allow-legacy-userdetails-fallback=true. É dívida MEDIUM para um fluxo canónico.

21. JWT TENANT-SCOPED
-------------------------------------------------------------------------------

Claims exactos:
- sub;
- iat/exp;
- roles globais CSV;
- userId;
- tokenType="TENANT";
- tenantId;
- tenantCode;
- tenantRoles List<String>;
- tenantUserStatus;
- platformAdmin Boolean;
- tenantAccessVersion, salvo bypass Platform Admin;
- tenantPermissionsUpdatedAt, se existir.

O contexto é suficiente para /api/tenant/**. O TenantResolver revalida Tenant
ATIVO e membership ATIVA em cada request normal; com require-access-version=true
compara a versão do token e devolve 401 TENANT_TOKEN_STALE quando diverge.

Matriz:

Tipo | Uso | Endpoints | Contexto | Troca
GLOBAL | login/pré-tenant | /auth/tenants, /auth/tenant/select | User/roles globais | sim, seleccionar Tenant
TENANT | operação | /tenant/** | User + Tenant + tenant roles/version | trocar ao mudar de Business

22. TENANT RESOLVER, CONTEXT E X-TENANT-ID
-------------------------------------------------------------------------------

TenantResolver prioriza JWT TENANT. Também suporta:
- X-Tenant-Id;
- X-Tenant-Code;
- auto-resolução para business user com exactamente uma linha TenantUser ATIVA;
- selecção explícita de Platform Admin.

Para JWT TENANT, X-Tenant-Id não é necessário. Para fluxos legacy/global, header
é uma alternativa e pode ser necessário quando existem várias memberships. O
contrato canónico recomendado é o JWT TENANT emitido pelo select, sem header.

TenantContextFilter coloca contexto por request e restaura/limpa em finally.
GET /auth/tenants é excluído exactamente por ser pré-tenant. O contexto vem de
claim/header/membership, nunca de BusinessAccount.

Risco: Platform Admin via POST select gera auditoria persistente; a selecção via
header é resolvida directamente e não chama auditContextAssumed. A garantia de
auditoria elevada é, portanto, PARCIAL.

23. ACCESS VERSION
-------------------------------------------------------------------------------

V18 criou tenant_user_access_versions com unique(tenant,user), access_version e
permissions_updated_at. TenantUsuarioService incrementa a versão ao criar,
alterar roles, suspender, reactivar ou remover membership.

TenantResolver exige claim e compara com a linha, usando 1 como fallback. Isso
rejeita token antigo após alterações realizadas pelo serviço tenant-admin.

Owner replacement e manager governance não chamam esse serviço porque não
sincronizam TenantUser. Consequentemente não invalidam tokens operacionais.

24. BUSINESSACCOUNT <-> TENANT
-------------------------------------------------------------------------------

Relação: tenants.business_account_id FK opcional para business_accounts.id,
introduzida em V72; índice idx_tenants_business_account. O provisioning canónico
passa businessAccountId ao template e exige a ligação ao regressar.

GET /auth/tenants poderia alcançar a Account via Tenant, mas não selecciona nem
expõe os seus campos. Isso é suficiente para contexto operacional; falta apenas
aplicar a policy de elegibilidade da Account se essa for a regra canónica.

25. VERTICAL E WORKSPACE ROUTING
-------------------------------------------------------------------------------

O provisioning recebe BusinessVertical CONSUMA_PONTO/CONSUMA_REST e deriva:
- CONSUMA_PONTO -> tenant.templateCode CONSUMA_PONTO, templateVersion 1;
- CONSUMA_REST -> tenant.templateCode CONSUMA_REST, templateVersion 1.

OperationalTemplatePolicy normaliza ambos para CONSUMA_PONTO_V1 ou
CONSUMA_REST_V1 e também aceita códigos já versionados. Logo templateCode é a
fonte canónica existente e suficiente para routing PONTO vs REST neste schema.

GET /auth/tenants não expõe campo vertical explícito, mas expõe templateCode.
A inferência é segura somente por allowlist dos códigos canónicos acima. Código
legacy/desconhecido deve levar a workspace não suportado, nunca fallback para
PONTO/REST. Um campo vertical explícito seria útil, mas não necessário.

26. SUBSCRIPTION
-------------------------------------------------------------------------------

GET resolve planoCodigo exclusivamente por SubscricaoEstado.ATIVA. Se não houver
subscrição activa, Tenant ainda aparece e planoCodigo é null. POST select e
TenantResolver também permitem acesso enquanto Tenant/membership estão activos.

Readiness exige Subscricao legacy ATIVA antes da activação inicial. Depois disso,
expiração/suspensão/cancelamento não afecta automaticamente Tenant, selector ou
token. É gap HIGH se a política comercial pretende bloquear operação.

27. PONTO COMPATIBILITY
-------------------------------------------------------------------------------

SIM no contrato de contexto. O template PONTO cria TENANT_OWNER ATIVO; POST
select emite o mesmo JWT TENANT; controllers /tenant e policies aceitam
TENANT_OWNER/TENANT_ADMIN nas operações administrativas e operacionais. Turno,
instituição, unidade, módulos e subscrição podem ser pré-condições de operações
específicas, mas não da autenticação/contexto.

Não existe E2E dedicado login->select->operação PONTO no conjunto seleccionado;
a compatibilidade foi provada por composição de provisioning PostgreSQL,
TenantScopedTokenAccessIT e policies/tests operacionais existentes.

28. REST COMPATIBILITY
-------------------------------------------------------------------------------

SIM no contrato de contexto. REST usa a mesma camada JWT/TenantResolver e cria
TENANT_OWNER ATIVO. Diferenças ficam nas capabilities/template, mesas, sessão,
turno, produção/KDS e recursos provisionados, não na selecção de Tenant.

Também aqui não houve um único E2E dedicado login->select->operação REST; a
conclusão é estática + testes transversais.

29. MULTI-BUSINESS
-------------------------------------------------------------------------------

Com Account A, Owner X e dois Businesses provisionados enquanto X permanece
responsavel, toTemplateRequest usa X em cada operação. Resultado: dois
TenantUsers TENANT_OWNER ATIVO e GET /auth/tenants retorna 2 Tenants.

Excepção: se X ganhar múltiplas roles num Tenant, GET pode retornar mais de duas
linhas/opções e select pode falhar pela inconsistência multi-role.

30. OWNER SCENARIOS
-------------------------------------------------------------------------------

CREATE_NEW completo:
User existe: SIM, criado no approval da Account.
BusinessAccount OWNER: SIM, OWNER ATIVO.
Tenant existe: SIM, após provisioning.
TenantUser: SIM, criado no provisioning.
Role/estado: TENANT_OWNER/ATIVO.
Após activar Account e Tenant, GET /auth/tenants: SIM.
POST select e tenant JWT: SIM, desde que membership seja singular.

EXISTING Account + EXISTING Owner + novo Business:
O Owner actual da Account é usado pelo provisioning e recebe TENANT_OWNER ATIVO
nesse novo Business. Businesses antigos só seguem o Owner que existia quando
foram provisionados; replacement não sincroniza retroactivamente.

31. MANAGER SCENARIO
-------------------------------------------------------------------------------

Adicionar Manager sozinho não muda GET /auth/tenants: resultado 0 para esse
Business se não houver TenantUser anterior. Não há propagação automática para
Businesses existentes ou futuros.

É parcialmente design: governance empresarial e acesso operacional são
conceitos separados e o provisioning exige escolha explícita. É gap de segurança
quando remove/suspende/rebaixa um Manager que já possui TenantUser, pois a
revogação não propaga nem invalida token.

32. OWNER REPLACEMENT SCENARIO
-------------------------------------------------------------------------------

Antes: Owner A vê e selecciona os Tenants onde é TENANT_OWNER; B não.
Depois do replace empresarial: A continua a ver/seleccionar; B continua sem ver
os antigos. TenantUser não muda. Novos Businesses passam a atribuir B.

O teste existente ownerReplacement_isAtomicAndManagerCannotBecomeOwner prova
apenas a invariância empresarial; não afirma TenantUser. O código confirma a
ausência total de sincronização.

33. SECURITY RISKS
-------------------------------------------------------------------------------

BLOCKER
- Nenhum para provar que o contrato base existe. Para activar recovery em
  produção, a falha multi-role deve ser tratada como bloqueante da entrega UI.

HIGH
- GAP-01: multi-role duplica GET e pode quebrar POST select.
- GAP-02: Owner replacement deixa antigo Owner operacional e novo Owner sem
  acesso aos Businesses existentes; tokens não são invalidados.
- GAP-03: remoção/suspensão de Manager empresarial não revoga TenantUser/token.
- GAP-04: Account ou subscrição inactiva não bloqueia selector/resolver se o
  Tenant continuar ATIVO.

MEDIUM
- selector declara JWT GLOBAL, mas não impõe tokenType;
- refresh JWT gera formato legacy e depende do fallback configurável;
- Platform Admin por header não tem a mesma auditoria explícita do POST select;
- principal é ordem técnica, não preferência canónica;
- não há teste E2E único CREATE_NEW -> login -> list -> select para PONTO/REST;
- User desactivado após emissão depende de flags strict/validate-user-active,
  ambas false por default no filtro.

LOW
- BusinessAccount não é exposta para agrupamento visual;
- vertical não é campo explícito, embora templateCode seja suficiente;
- tenantId é ID interno sequencial exposto no contrato autenticado;
- mensagens/HTTP de Tenant inexistente/inactivo em select são 400, não uma
  taxonomia específica;
- roles globais e tenant roles coexistem em authorities e exigem guards atentos.

Riscos frontend:
- fallback automático para o primeiro Tenant é inseguro e principal não é uma
  preferência persistida;
- localStorage expõe tokens a XSS;
- duas tabs podem colidir ao partilhar um único contexto tenant persistido;
- troca de Tenant com requests in-flight pode misturar respostas de contexto;
- o interceptor histórico não enviava tenantToken nas APIs operacionais.
