PROMPT 40.1 — LOCALREF + ENCADEAMENTO SEGURO OFFLINE + PAYLOAD LIMITS
Data: 2026-05-23

1) RESUMO EXECUTIVO
- Adicionado suporte a `localRef`/`dependsOn` em comandos do endpoint `/device/offline-sync/batch`, permitindo encadeamento seguro no mesmo batch:
  - `CREATE_PEDIDO_POS` → `CREATE_ORDEM_PAGAMENTO_MANUAL` via `pedidoClientRequestId`
  - `CREATE_ORDEM_PAGAMENTO_MANUAL` → `CONFIRM_MANUAL_PAYMENT` via `ordemPagamentoClientRequestId`
- Implementada resolução de dependências:
  - intra-batch (mapa localRef/clientRequestId → entidade real);
  - entre batches (lookup em `device_offline_commands` do mesmo tenant+device).
- Implementados limites explícitos de payload (por comando e por batch), limite de itens por pedido offline e hardening de dependências (ciclo, forward-ref bloqueado por default, profundidade máxima).
- Idempotência do Prompt 40 preservada: `tenantId + deviceId + clientRequestId` + `payloadHash`.

7) SERVICE ALTERADO (CORE)
- `src/main/java/com/restaurante/service/device/offline/DeviceOfflineSyncService.java`
  - validações novas:
    - `clientRequestId` único no batch;
    - `localRef` único no batch;
    - `maxBatchPayloadBytes` e `maxCommandPayloadBytes` (calculado no backend via JSON canônico em UTF-8);
    - `maxPedidoItems` para `CREATE_PEDIDO_POS`;
    - dependências:
      - forward refs bloqueadas por default;
      - ciclo bloqueado;
      - profundidade máxima bloqueada.
  - resolução de dependência:
    - intra-batch e entre batches (mesmo tenant+device).
  - “payload rewrite”:
    - injeta `pedidoId` quando existir `pedidoClientRequestId` resolvido;
    - injeta `ordemPagamentoId` quando existir `ordemPagamentoClientRequestId` resolvido;
    - valida `entityType` esperado (PEDIDO/ORDEM_PAGAMENTO).
  - persistência de `result_json` passou a armazenar wrapper:
    - `{ "result": <resultado_handler>, "resolvedRefs": {...} }`
    - resposta HTTP continua retornando o `result` “plano” (compatível com Prompt 40).

8) HANDLERS
- Não houve redesign de domínio; o encadeamento foi habilitado por:
  - resolução no service + injeção de IDs no payload antes de chamar os handlers existentes.

9) COMO LOCALREF FUNCIONA (RESUMO)
- Por default, o POS usa o próprio `clientRequestId` como `localRef`.
- Comandos dependentes referenciam `*ClientRequestId` no payload.
- O servidor resolve:
  - primeiro no mapa do batch;
  - senão no histórico `device_offline_commands` do mesmo device.

10) SEGURANÇA (CROSS-TENANT / CROSS-DEVICE)
- Resolução histórica exige:
  - mesmo `tenantId` e mesmo `deviceId` (chave do repository).
- Não permite device A usar localRef de device B.
- Não permite tenant A usar localRef de tenant B.
- Mesmo com ref resolvido, o processamento revalida policies/capabilities/turno no domínio (reuso de services).

14) ITENS NÃO COBERTOS (POR DESIGN)
- Forward refs habilitadas por default (mantido false).
- Rollback do batch.
- AppyPay/OTP offline.
- Dependências múltiplas complexas por comando (além do caso principal).

15) DECISÕES TÉCNICAS
- Preferido “payload rewrite” no `DeviceOfflineSyncService` para manter handlers simples e reaproveitar os services existentes.
- Persistência de wrapper em `result_json` para manter rastreabilidade de resolução, mas resposta HTTP preserva compatibilidade (retornando `result` plano).
