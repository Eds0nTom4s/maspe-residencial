1) RESUMO EXECUTIVO
- Implementado o primeiro elo real entre o domínio SaaS e o domínio operacional: `Tenant 1:N Instituicao`.
- `Instituicao` agora possui `tenant_id` obrigatório (NOT NULL) e relacionamento JPA `ManyToOne` com `Tenant`.
- Criada migration Flyway `V3__link_instituicao_to_tenant.sql` com backfill compatível para bases existentes:
  - cria Tenant LEGACY (tenant_code=LEGACY) se necessário
  - vincula instituições existentes ao tenant LEGACY
  - garante integridade com FK + NOT NULL
- Adicionados métodos tenant-aware no `InstituicaoRepository`.
- Adicionados testes de integração (PostgreSQL/Testcontainers) provando Tenant 1:N e NOT NULL em `instituicoes.tenant_id`.
- Fluxos legados continuam existindo (getInstituicaoAtiva/findFirstByAtivaTrue não removidos).

2) ESTADO ANTES
- Tenant Core (V2) existia, mas `Instituicao` ainda era “órfã” do ponto de vista SaaS.
- Domínio operacional seguia single-tenant via “instituição ativa”.

3) ALTERAÇÕES DE DOMÍNIO (JPA)
src/main/java/com/restaurante/model/entity/Instituicao.java
- Adicionado:
  - `@ManyToOne(fetch = LAZY, optional = false)`
  - `@JoinColumn(name="tenant_id", nullable=false)`
- Index:
  - `idx_instituicao_tenant (tenant_id)`

4) MIGRATION FLYWAY V3
src/main/resources/db/migration/V3__link_instituicao_to_tenant.sql
- Passos:
  1) adiciona coluna `tenant_id` (nullable inicialmente)
  2) cria índice `idx_instituicao_tenant`
  3) cria Tenant LEGACY se não existir
  4) cria subscrição ATIVA para o tenant LEGACY no plano PILOTO (mantém coerência do domínio SaaS)
  5) backfill: seta `instituicoes.tenant_id` para LEGACY
  6) impõe `NOT NULL` + FK `instituicoes(tenant_id) -> tenants(id)`

5) COMPATIBILIDADE (LEGADO / DEV)
src/main/java/com/restaurante/config/DataSeeder.java
- No profile `dev`, o seed de Instituicao agora:
  - cria (se necessário) o Tenant LEGACY + subscrição ATIVA no PILOTO
  - associa a Instituicao seed ao tenant LEGACY
