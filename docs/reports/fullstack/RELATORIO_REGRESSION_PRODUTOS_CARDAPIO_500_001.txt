GET http://localhost:8081/api/tenant/produtos?size=100

retornava 500 Internal Server Error no backend. A UI apenas exibiu a falha recebida do endpoint tenant de produtos.

5. Stacktrace backend

Classe de erro principal:

org.springframework.http.converter.HttpMessageNotWritableException:
Could not write JSON: failed to lazily initialize a collection of role:
com.restaurante.model.entity.Produto.imagensGaleria: could not initialize proxy - no Session

Causa interna:

org.hibernate.LazyInitializationException:
failed to lazily initialize a collection of role:
com.restaurante.model.entity.Produto.imagensGaleria: could not initialize proxy - no Session

Cadeia de serializacao:

ApiResponse["data"]
-> PageImpl["content"]
-> ProdutoResponse["imagensGaleria"]

Classificacao da causa:
J. Outro: conflito de mapeamento ORM/DTO com lazy collection serializada fora da sessao Hibernate.

6. Causa raiz

ProdutoService.mapToResponse atribuía diretamente produto.getImagensGaleria() ao ProdutoResponse.

Depois do merge, Produto.imagensGaleria estava mapeado como @ElementCollection(fetch = LAZY) sobre a tabela produto_imagens. O objeto Page<ProdutoResponse> era montado dentro da transacao read-only, mas a colecao lazy era apenas referenciada no DTO. Quando Jackson serializava a resposta HTTP fora do escopo da sessao Hibernate, tentava inicializar Produto.imagensGaleria e disparava LazyInitializationException.

Durante o teste de regressao foi confirmada uma segunda incompatibilidade relacionada: produto_imagens ja e a tabela canonica da entidade ProdutoImagem, com colunas obrigatorias tenant_id, produto_id, url e ordem. Persistir galeria via ElementCollection em Produto tentava inserir linhas sem tenant_id/url canonicos, gerando violacao de schema. Portanto a correcao nao poderia ser apenas abrir a transacao ou mascarar a lista; era necessario remover a dupla responsabilidade ORM.

7. Relacao com V77

Migration auditada: V77__store_schema_reconciliation.sql

Resultado no banco Docker local:

flyway_schema_history:
77|store schema reconciliation|t

Logs do backend Docker local:

Current version of schema "public": 77

Conclusao:

A V77 estava aplicada com sucesso. O problema nao era migration ausente. A regressao surgiu da compatibilidade entre o schema consolidado de produto_imagens e o mapeamento Produto.imagensGaleria como ElementCollection, competindo com a entidade ProdutoImagem.

8. Correcao realizada

Backend:

- Produto.imagensGaleria deixou de ser persistido como ElementCollection e passou a ser @Transient.
- ProdutoService passou a resolver imagensGaleria pela entidade canonica ProdutoImagemRepository.findByTenantIdAndProdutoIdOrderByOrdemAsc.
- ProdutoResponse continua expondo imagensGaleria, mas recebe uma lista materializada de URLs antes da serializacao JSON.
- StoreMapper recebeu a mesma resolucao por ProdutoImagemRepository para reduzir o risco Store/Socio apontado no merge anterior.
- Foi preservado tenant isolation usando tenantId do produto antes de buscar imagens.
- Nao houve fallback silencioso para lista vazia quando o erro real ocorre; apenas produtos sem galeria retornam lista vazia controlada.

Teste de regressao:

- Adicionado TenantProdutoControllerIT.
- O teste cria tenant, usuario tenant-scoped, categoria, produto e galeria pela entidade ProdutoImagem.
- Executa GET /tenant/produtos?size=100 com token tenant-scoped.
- Valida HTTP 200, success true, produto esperado e imagensGaleria serializada.
