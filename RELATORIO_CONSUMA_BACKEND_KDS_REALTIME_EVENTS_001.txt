1. Objetivo

Adicionar notificacoes realtime tenant-scoped para KDS usando a infraestrutura WebSocket/STOMP
ja existente, sem alterar UI, mantendo REST KDS funcional e sem vazamento cross-tenant.

2. Decisao tecnica

* mecanismo usado:
  - WebSocket + STOMP existente no backend
  - endpoint SockJS: /api/ws
  - broker: /topic e /queue
* SSE:
  - NAO usado, porque o backend ja possuia WebSocket/STOMP e broker simples configurados
* topicos publicados:
  - /topic/tenant/{tenantId}/kds
  - /topic/tenant/{tenantId}/kds/unidade/{unidadeProducaoId}
* seguranca dos topicos:
  - CONNECT exige Authorization Bearer
  - SUBSCRIBE para /topic/tenant/{tenantId}/... exige token tenant-scoped
  - token global sem tenant selecionado e bloqueado
  - cross-tenant e bloqueado no interceptor STOMP

3. Contrato de evento

* DTO:
  - KdsRealtimeEventResponse
* campos:
  - eventId
  - tenantId
  - eventType
  - subPedidoId
  - pedidoId
  - pedidoNumero
  - unidadeProducaoId
  - unidadeProducaoNome
  - statusAnterior
  - statusAtual
  - version
  - occurredAt
* tipos:
  - SUBPEDIDO_CREATED
  - SUBPEDIDO_STATUS_CHANGED
  - SUBPEDIDO_STARTED
  - SUBPEDIDO_READY
  - SUBPEDIDO_DELIVERED
  - SUBPEDIDO_CANCELLED
  - KDS_REFRESH_REQUIRED

4. Pontos de publicacao

* criacao de subpedido:
  - PublicQrPedidoService => SUBPEDIDO_CREATED
  - PedidoService => SUBPEDIDO_CREATED
  - SubPedidoService => SUBPEDIDO_CREATED
* transicoes KDS:
  - KdsOperationsService.iniciarPreparo => SUBPEDIDO_STARTED
  - KdsOperationsService.marcarPronto => SUBPEDIDO_READY
  - KdsOperationsService.entregar => SUBPEDIDO_DELIVERED
* sem publicacao indevida:
  - transicoes invalidas/conflitos 409 nao publicam evento de sucesso
* consistencia transacional:
  - eventos sao enfileirados e publicados apos commit via TransactionalEventListener(AFTER_COMMIT)
  - falha de envio realtime e registrada em log e nao quebra a operacao principal

5. Seguranca tenant-scoped

* CONNECT sem token:
  - bloqueado pelo JwtChannelInterceptor
* SUBSCRIBE sem autenticacao:
  - bloqueado
* SUBSCRIBE com token global:
  - bloqueado para topicos tenant-scoped
* SUBSCRIBE cross-tenant:
  - bloqueado quando destination tenantId difere do tenantId do token
* validacao automatizada:
  - KdsRealtimeEventsWebSocketIT cobre CONNECT, token global, cross-tenant e subscribe valido
* validacao runtime REST:
  - GET /api/tenant/kds/subpedidos sem Authorization => 401
  - GET /api/tenant/kds/subpedidos/4 com token tenant 2 para subpedido tenant 3 => 404

* a validacao runtime WebSocket com cliente externo ficou limitada pela instalacao quebrada do wscat no host local;
  a prova funcional STOMP ficou coberta por teste de integracao.
