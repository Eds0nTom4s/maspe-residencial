5.2) PaymentMethodPolicyRolloutWorkerService
- Respeito a cancelRequested:
  - se cancelRequested ou status CANCEL_REQUESTED: não processa novos itens, cancela itens PENDING, e finaliza CANCELLED quando não houver RUNNING.
- Backoff por item:
  - claim só pega itens PENDING com nextRetryAt <= now e terminal_failure=false
  - ao falhar item e não atingir maxAttempts: agenda nextRetryAt com backoff exponencial.
  - ao atingir maxAttempts: marca FAILED + terminalFailure=true
- Stale lock recovery:
  - recoverStaleRolloutLocks(): limpa locked_at/locked_by para rollouts RUNNING/CANCEL_REQUESTED com lock antigo e audita PAYMENT_POLICY_ROLLOUT_STALE_LOCK_RECOVERED
  - recoverStaleRunningItems(): itens RUNNING com last_locked_at antigo voltam para PENDING (com backoff) ou FAILED terminal
- Rate-limit de progress events:
  - PAYMENT_POLICY_ROLLOUT_PROGRESS_UPDATED só emite por delta percentual mínimo OU intervalo mínimo, e grava last_progress_event_at/percent

7) COMO O CANCELAMENTO FUNCIONA
- PENDING:
  - status=CANCELLED
  - itens status=PENDING -> CANCELLED (skippedReason=CANCELLED_BY_USER)
- RUNNING:
  - status=CANCEL_REQUESTED
  - cancelRequested=true
  - worker passa a:
    - não claimar novos itens
    - cancelar itens PENDING
    - finalizar CANCELLED quando não existirem itens RUNNING
- Cancelamento não é rollback: não desfaz CREATED/UPDATED.

8) COMO O BACKOFF FUNCIONA
- Ao falhar um item:
  - attempts já é incrementado no claim do item
  - se attempts < maxAttempts:
    - status=PENDING
    - nextRetryAt=now + delay exponencial (initialBackoffSeconds * multiplier^(attempts-1), limitado a maxBackoffSeconds)
  - se attempts >= maxAttempts:
    - status=FAILED
    - terminalFailure=true
- Worker só pega itens quando nextRetryAt <= now.

9) COMO O STALE LOCK RECOVERY FUNCIONA
- Rollout:
  - se lockedAt < now - staleLockTimeoutSeconds:
    - lockedAt/lockedBy limpos
    - staleRecoveryCount++
    - evento PAYMENT_POLICY_ROLLOUT_STALE_LOCK_RECOVERED
- Item RUNNING:
  - se lastLockedAt < now - staleItemTimeoutSeconds:
    - attempts++
    - status=PENDING (com backoff) ou FAILED terminal
    - errorCode=STALE_LOCK_RECOVERED

10) RATE-LIMIT DE AUDITORIA DE PROGRESSO
- Emite PAYMENT_POLICY_ROLLOUT_PROGRESS_UPDATED apenas se:
  - progressPercent avançou >= progressEventMinPercentDelta; OU
  - passou >= progressEventMinIntervalSeconds desde o último evento.
- Eventos finais continuam sendo emitidos normalmente (COMPLETED/PARTIAL_FAILED/FAILED/CANCELLED).

11) SEGURANÇA / RBAC (TENANT-AWARE)

- Cross-tenant: rollout/template/unidade inválidos retornam 404/erro.

19) DECISÕES TÉCNICAS TOMADAS
- Backoff e retries controlados por item (melhor tolerância a falhas sem travar o rollout).
- Stale recovery automático para evitar rollouts travados por crash do worker.
- Rate-limit de progresso para reduzir ruído de auditoria sem perder visibilidade.
- Cancelamento como interrupção (não rollback) para manter consistência operacional.

20) RISCOS RESTANTES
- Sem Docker no host, a suite it-postgres não é executada integralmente localmente/CI.
- Stale recovery pode reprocessar item cuja policy já foi aplicada; mitigação via detecção POLICY_ALREADY_MATCHES.
