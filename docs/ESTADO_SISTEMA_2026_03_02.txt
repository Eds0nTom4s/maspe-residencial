1.4  MODELO DE NEGÓCIO CLARIFICADO E APLICADO NO CÓDIGO
──────────────────────────────────────────────────────────────────────────────
Fundo de Consumo:
  - É um MÉTODO DE PAGAMENTO pré-pago, não um produto nem uma conta de cliente.
  - O QR Code (tokenPortador) é o identificador universal do fundo.
  - Fundo identificado: tem clienteId + token gerado automaticamente.
  - Fundo anónimo: sem clienteId, token = QR Code da mesa (UnidadeConsumo.qrCode).
  - Após criação: AMBOS operam exclusivamente pelo token. O garçom nunca precisa
    saber se o fundo é identificado ou anónimo — lê o QR, usa o token, acabou.

Mesa (UnidadeConsumo):
  - Pre-registada pelo admin. Não tem dependência directa do cliente.
  - O QR Code é o elo físico entre a mesa e o sistema.
  - Fluxo: cliente senta → garçom lê QR da mesa → token identificado →
    cria pedido ou fundo de consumo.

2.7  FLUXO OPERACIONAL COMPLETO (como o sistema funciona no dia-a-dia)
──────────────────────────────────────────────────────────────────────────────
  1. SETUP (admin, feito uma vez):
     → Criar UnidadeAtendimento (ex: "Restaurante Principal")
     → Criar UnidadeConsumo (ex: "Mesa 7") com QR Code gerado
     → Criar Cozinhas (ex: "Cozinha Principal", "Bar")
     → Associar produtos a cozinhas

  2. ABERTURA DE MESA:
     → Garçom lê QR da mesa → obtém token da UnidadeConsumo
     → Cria Pedido associado à mesa

  3. LANÇAMENTO DE ITENS:
     → Pedido criado → sistema distribui automaticamente os itens
       pelos SubPedidos por cozinha (ex: bebidas → Bar, pratos → Cozinha)

  4. COZINHA:
     → Cozinheiro vê SubPedidos ativos em /api/subpedidos/cozinha/{id}/ativos
     → Actualiza: RECEBIDO → EM_PREPARO → PRONTO
     → Garçom entrega e marca como ENTREGUE

  5. PAGAMENTO — três métodos possíveis:
     a) FUNDO DE CONSUMO (pré-pago):
        → Garçom lê QR do fundo → POST /{token}/recarregar (se necessário)
        → Sistema debita automaticamente quando pedido é fechado
     b) APPYPAY (referência multibanco/móvel):
        → Sistema gera referência → cliente paga → callback confirma
     c) PÓS-PAGO:
        → Disponível apenas se ativado pelo ADMIN em /api/configuracoes-financeiras
        → Sujeito a limite de risco configurável

  6. ENCERRAMENTO:
     → Pedido financeiramente fechado → SubPedidos concluídos
     → Mesa disponível novamente

3.1  ENDPOINTS EM FALTA
──────────────────────────────────────────────────────────────────────────────
[CRÍTICO] ClienteController — gestão de clientes via REST
  Problema : o ClienteService existe e tem todas as operações, mas não há
             controller dedicado. O único acesso a clientes é via /api/auth.
             O frontend (painel admin) não tem como listar, editar ou gerir
             clientes sem este controller.
  Solução  : criar ClienteController com:
               GET  /api/clientes            → listar (paginado, GERENTE/ADMIN)
               GET  /api/clientes/{id}       → buscar por ID
               GET  /api/clientes/telefone/{tel} → buscar por telefone
               PUT  /api/clientes/{id}       → actualizar perfil
               DELETE /api/clientes/{id}    → desactivar (ADMIN)

[CRÍTICO] PagamentoController — initiating de pagamento pelo staff
  Problema : PagamentoCallbackController só processa retornos da AppyPay.
             Não existe endpoint para o STAFF iniciar um pagamento
             (gerar referência AppyPay para uma mesa).
  Solução  : criar PagamentoController com:
               POST /api/pagamentos/gerar-referencia
                    → body: { pedidoId, metodo: "APPYPAY" }
                    → retorna referência + QR Code AppyPay para o cliente
               GET  /api/pagamentos/{pedidoId} → estado do pagamento
               GET  /api/pagamentos/pedido/{id} → histórico de pagamentos

[CRÍTICO] Endpoint de fechamento de conta (checkout)
  Problema : não existe um endpoint explícito para "fechar a conta" de uma
             mesa — ou seja, encerrar o pedido, cobrar, e libertar a mesa.
             O PedidoFinanceiroService tem lógica interna mas não está exposto.
  Solução  : adicionar em PedidoController:
               POST /api/pedidos/{id}/fechar
                    → body: { metodoPagamento, tokenFundo (se FUNDO_CONSUMO) }
                    → orquestra pagamento + encerramento do pedido + liberta mesa

[ALTO] Endpoint de débito do Fundo de Consumo por pedido
  Problema : FundoConsumoService.debitarPorToken(token, pedidoId, valor) existe
             mas não está exposto via REST. O frontend não tem como disparar
             o débito manualmente se a integração automática falhar.
  Solução  : adicionar em FundoConsumoController:
               POST /api/fundos/{token}/debitar
                    → body: { pedidoId, valor }
                    → útil para recuperação manual de falhas

[ALTO] RegisterController / endpoint de criação de staff
  Problema : JwtAuthController tem um endpoint de register, mas não há
             um fluxo completo para o admin criar atendentes/gerentes com
             role específico.
  Solução  : expandir UserController ou criar endpoint dedicado:
               POST /api/usuarios/atendente  (ADMIN)
               POST /api/usuarios/gerente    (ADMIN)

3.2  INTEGRAÇÕES EM FALTA OU INCOMPLETAS
──────────────────────────────────────────────────────────────────────────────
[CRÍTICO] Integração automática Fundo de Consumo ↔ Pedido
  Problema : quando um pedido é criado com método FUNDO_CONSUMO, o sistema
             deveria debitar automaticamente o fundo. Não é claro se
             PedidoService.criar() chama FundoConsumoService.debitarPorToken().
             Se não chamar, o débito nunca acontece.
  Acção    : verificar PedidoService.criar() e PedidoFinanceiroService.
             Se não integrado, implementar a chamada.

[ALTO] WebSocket para actualizações em tempo real
  Problema : WebSocketNotificacaoService existe mas não é claro se está
             activamente a emitir eventos para:
               - cozinha recebe novo sub-pedido em tempo real
               - garçom é notificado quando prato fica pronto
               - painel do admin vê métricas a actualizar
  Acção    : verificar configuração do WebSocket e testar fluxo end-to-end.

[MÉDIO] Estorno automático AppyPay
  Problema : se um pedido AppyPay pago for cancelado, o sistema deve chamar
             a API de estorno da AppyPay. Não confirmado se está implementado.
  Acção    : verificar PagamentoCallbackService e fluxo de cancelamento.

3.3  FUNCIONALIDADES DE NEGÓCIO EM FALTA
──────────────────────────────────────────────────────────────────────────────
[ALTO] Reservas de mesa
  Estado  : não existe entidade nem endpoints para reservas.
  Impacto : se o negócio aceitar reservas (hotéis, jantares especiais),
            o sistema não suporta.

[ALTO] Relatórios financeiros
  Estado  : DashboardController tem stats básicas. Não há:
            - relatório de receita por período
            - relatório de produtos mais/menos vendidos com totais
            - relatório de uso de Fundo de Consumo
            - exportação CSV/PDF
  Acção   : criar RelatóriosController com endpoints dedicados.

[MÉDIO] Gestão de stock / inventário
  Estado  : Produto tem campos de preço e disponibilidade, mas não há
            controlo de stock.
  Impacto : sistema não avisa quando produto esgota.

[MÉDIO] Histórico de pedidos por mesa (UnidadeConsumo)
  Estado  : não existe endpoint GET /api/unidades-consumo/{id}/pedidos.
  Impacto : o frontend não consegue mostrar o histórico de uma mesa específica.

[MÉDIO] Transferência de pedido entre mesas
  Estado  : não existe endpoint para mover um pedido de uma mesa para outra.

[BAIXO] Desconto e promoções
  Estado  : não existe entidade de desconto. ItemPedido não tem campo de desconto.

[BAIXO] Impressão de recibo / comanda
  Estado  : não existe endpoint que retorne um recibo formatado para impressão.
  Nota    : CozinhaController pode já ter algo para comanda — verificar.

3.4  QUALIDADE E TESTES
──────────────────────────────────────────────────────────────────────────────
[ALTO] Cobertura de testes automatizados
  Estado  : estrutura de testes existe (src/test/), mas cobertura desconhecida.
  Risco   : código financeiro (FundoConsumo, PedidoFinanceiro, AppyPay)
            sem testes unitários é risco crítico em produção.
  Acção   : implementar pelo menos:
            - FundoConsumoServiceTest (débito, saldo insuficiente, idempotência)
            - PedidoFinanceiroServiceTest
            - AppyPay callback handling test

[MÉDIO] Testes de integração E2E
  Estado  : ficheiro PERFIL_ESPECIALISTA_TESTES_E2E.txt existe → indica que
            havia intenção. Verificar se testes foram implementados.

[MÉDIO] DebugController em produção
  Risco   : DebugController não deve estar activo em produção.
  Acção   : garantir @Profile("dev") ou desabilitar via configuração.

3.5  CONFIGURAÇÃO E OPERAÇÕES
──────────────────────────────────────────────────────────────────────────────
[ALTO] Migração de base de dados (Flyway / Liquibase)
  Estado  : usa data.sql (script estático). Em produção, alterações de schema
            requerem migrações versionadas.
  Acção   : adoptar Flyway com scripts V1__init.sql, V2__add_fundo.sql, etc.

[ALTO] Variáveis sensíveis em application.properties
  Risco   : credenciais AppyPay, TelcoSMS, JWT secret podem estar hardcoded.
  Acção   : mover para variáveis de ambiente / vault. Ver DEPLOYMENT.md.

[MÉDIO] Rate limiting nos endpoints públicos
  Estado  : /api/auth/solicitar-otp é público. Sem rate limit → risco de
            abuso (custo de SMS).
  Acção   : adicionar rate limiting (Bucket4j ou similar).

[BAIXO] Health check e métricas (Spring Actuator)
  Estado  : não confirmado se Spring Actuator está configurado.
  Acção   : expor /actuator/health para monitorização em produção.
