5) Scope resolver criado
- `src/main/java/com/restaurante/service/producao/ProducaoScopeResolver.java`
  - Usuário humano: se houver 1 unidade ativa no tenant, usa-a; se houver múltiplas, retorna `EXPLICIT_REQUIRED` com opções (sem persistir seleção nesta fase).
  - DevicePrincipal: resolve preferencialmente por `tenantId + unidadeAtendimentoId`; fallback seguro por `instituicaoId` e, se necessário, por “single active unit” do tenant; ambiguidade retorna 409 `DEVICE_PRODUCTION_UNIT_AMBIGUOUS`.

  - `findByTenantIdAndUnidadeAtendimentoIdAndAtivoTrueOrderByOrdemAsc(...)`
- `src/main/java/com/restaurante/repository/SubPedidoRepository.java`
  - Paginação em 2 passos (IDs paginados + fetch detalhes por `IN (:ids)`) para evitar paginação com fetch-join em coleções.
  - Projeção de métricas (rows com timestamps/status) para cálculo eficiente por período.

7) Estratégia de paginação/lookback
- Sempre paginado.
- `size` acima do máximo é limitado automaticamente.
- Se `de/ate` não informado, aplica lookback padrão de produção.

8) Estratégia de métricas
- MVP calcula médias com período limitado (lookback), ignorando timestamps ausentes:
  - `tempoMedioAteIniciarSegundos`
  - `tempoMedioAteProntoSegundos`
  - `tempoMedioTotalPreparacaoSegundos`
- Sem agregação SQL avançada/particionamento nesta fase.

9) Estratégia user/device
- Usuário humano: tenant vem do `TenantContext`.
- DevicePrincipal: tenant vem do próprio principal; exige `DeviceCapability.VIEW_PRODUCTION` para leitura.
- Device não pode usar endpoint geral `/tenant/producao/subpedidos` nem `/tenant/producao/metricas`.

10) RBAC/capabilities aplicadas
- `TENANT_OWNER|TENANT_ADMIN|TENANT_OPERATOR`: listagem geral e métricas gerais.
- `TENANT_KITCHEN`: apenas `/minha-unidade/*` (unidade/subpedidos/métricas).
- `TENANT_FINANCE` e `TENANT_CASHIER`: bloqueados em produção nesta fase.
- Device: permitido em `/minha-unidade/*` somente com `VIEW_PRODUCTION`.

15) Decisões técnicas tomadas
- “Minha unidade” para usuário humano não cria vínculo User→UnidadeProducao nesta fase; retorna `EXPLICIT_REQUIRED` quando múltiplas unidades exigirem escolha.
- Device scope: tenta por `unidadeAtendimentoId` primeiro; fallback seguro por instituição/tenant para suportar unidades default legadas sem `unidade_atendimento_id`.
- Paginação de KDS: 2-step (page de IDs + fetch por IN) para evitar problemas clássicos de JPA com paginação + coleções.

16) Riscos restantes
- Se houver múltiplas unidades ativas por instituição/tenant e nenhuma regra/código padrão (“GERAL”) configurado, dispositivos podem cair em ambiguidade (409) até que o tenant padronize o vínculo UA→UnidadeProducao ou defina códigos.
- Métricas por Java em período curto é OK para piloto, mas deve migrar para agregações SQL/índices adicionais conforme volume crescer.
