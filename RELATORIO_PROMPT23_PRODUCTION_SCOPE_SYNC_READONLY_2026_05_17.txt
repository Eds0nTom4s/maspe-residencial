2) single active unit no tenant
3) múltiplas → EXPLICIT_REQUIRED

Ordem (device):
1) `DevicePrincipal.unidadeProducaoId`
2) vínculo por `unidadeAtendimentoId`
3) fallback seguro por instituição/tenant se houver “single active”

17) Decisões técnicas tomadas
- Persistência leve (1 registro por tenant+user) para seleção de unidade, sem criar UI nem modelo complexo user→unidades.
- Preferência direta por device via `unidade_producao_id`, mantendo `unidade_atendimento_id` para compatibilidade.
- Sync incremental por `updatedSince` (simples e suficiente para MVP), deixando ETag para etapa posterior.

18) Riscos restantes
- Sync de mesas calcula ocupação via query por mesa (N queries). Aceitável para piloto, mas precisa otimização (batch) quando volume crescer.
- `updatedAt` depende do auditing JPA; se algum fluxo não atualizar corretamente, o sync incremental pode omitir mudanças (fallback: snapshot completo).
